Host-level WordPress stacks get messy fast

Traditional server tooling installs Nginx, PHP, MariaDB, and Redis directly on the host. That works, until it doesn't. WPFY replaces all of it with one isolated Docker Compose stack per site.

The old way
  • x
    Shared host packagesOne PHP upgrade or package conflict can affect every site on the server.
  • x
    Multi-site sprawlSites share processes, users, and config; cleanly removing one site is harder than it should be.
  • x
    Ad-hoc workflowsSSL, backups, restore, diagnostics, and SFTP end up as one-off scripts and tribal knowledge.
  • x
    Hand-built stacksCompose files, proxies, and certificates assembled by hand for every site.
The WPFY way
  • ✓
    Docker-firstNo host-level Nginx, PHP, MariaDB, or Redis. The host stays clean.
  • ✓
    Per-site isolationSites never share PHP, database, cache, writable volumes, or networks.
  • ✓
    CLI-driven workflowsCreate, SSL, backup, restore, diagnostics, and SFTP are retry-safe commands.
  • ✓
    Safer SSLCertificates issue only after a DNS/IP preflight confirms the domain points at your server.

Installer

Ubuntu installer

One scripted install on a fresh VPS: Docker Engine, the wpfy CLI, and the shared runtime, with staged steps and dry-run support.

Isolation

Per-site Compose stacks

Every site is its own Docker Compose project with a private network, volumes, and database. Nothing is shared.

Routing

Traefik edge proxy

A single shared Traefik instance routes traffic to each site's containers, with no broad write access to site data.

SSL

SSL DNS/IP preflight

Let's Encrypt issuance is opt-in and gated: wpfy verifies DNS resolution and public IP match before requesting a certificate.

WordPress

WordPress provisioning

Full WordPress setup per site, with wp-cli available inside each site's container for plugin, theme, and content work.

Safety

Backup & restore

Create per-site backup archives and restore them with a command. Backups are kept private, never world-readable.

Health

Diagnostics

Structured checks across Docker, Traefik, and every managed site, plus per-site status, info, and log inspection.

Access

SFTP lifecycle

Enable, inspect, and manage per-site SFTP access so each site gets its own scoped file transfer entry point.

From fresh VPS to managed WordPress in three steps

1

Install WPFY

Run the installer on a fresh Ubuntu VPS. It sets up Docker and the wpfy CLI, then verify with:

$ wpfy --help
2

Create a site

Provision an isolated WordPress stack (Compose project, network, database, and volumes) in one command:

$ wpfy site create example.com --wp
# add -le for SSL with DNS/IP preflight
3

Manage from the CLI

SSL, backups, restore, diagnostics, and SFTP are all first-class commands:

$ wpfy site status example.com
$ wpfy site backup example.com
$ wpfy debug

Use cases

Host a fleet of client sites

Each site is its own Compose stack with a private network, database, and volumes. Hand every client scoped SFTP access and keep the server yours.

How it works 
$ wpfy site create client-a.com --wp -le
$ wpfy site create client-b.com --wp --php 8.3
$ wpfy sftp --enable client-a.com

Migrate and recover without fear

Archive any site to a private backup and bring it back with one command. SSL only issues after DNS provably points at your server.

Security model 
$ wpfy site backup example.com
$ wpfy site restore example.com example-backup.tar.gz
$ wpfy site ssl example.com
