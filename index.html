Why Docker-first

Host-level WordPress tooling installs shared packages on the server. WPFY replaces that with one isolated Docker Compose project per site, designed to reduce cross-site blast radius.

Host-level stacks

Shared infrastructure

  • ×
    Shared host packagesOne PHP upgrade or package conflict can affect every site on the server.
  • ×
    Shared PHP and databasesSites compete for one runtime and one database server.
  • ×
    Broad blast radiusProcesses, users, and credentials overlap across sites.
  • ×
    Ad-hoc workflowsSSL, backups, and diagnostics become one-off scripts.
WPFY

Per-site Compose projects

  • ✓
    No host-level Nginx, PHP, MariaDB, or RedisThe Ubuntu host stays clean; services run in containers.
  • ✓
    Per-site PHP selectionRun PHP 7.4 through 8.4 side by side where supported.
  • ✓
    Private per-site networkDedicated database, volumes, credentials, and Unix UID per site.
  • ✓
    Repeatable commandsCreate, SSL, backup, restore, and remove sites with retry-safe CLI workflows.

Architecture

Internet traffic enters a shared Traefik edge proxy, routes across the WPFY network, and terminates in isolated per-site Compose projects.

Command surface

Flat operator commands are primary where exact equivalents exist. Grouped wpfy site and wpfy stack commands remain for compatibility.

Provision sites and certificates

wpfy run example.com --wp
wpfy site ssl example.com --letsencrypt
wpfy site list

Operate runtimes and configuration

wpfy up example.com
wpfy down example.com
wpfy exec example.com app -- php -v
wpfy config example.com
wpfy refresh example.com --restart

Back up, restore, and retain

wpfy backup example.com
wpfy backup prune example.com --keep 7
wpfy backup schedule daily --time 02:30 --s3
wpfy restore example.com --latest
wpfy backup edge

Inspect health, logs, and security

wpfy healthcheck all
wpfy debug example.com
wpfy log show example.com --nginx -f
wpfy motd
wpfy secure example.com

Integrate WordPress, access, and services

wpfy wp example.com plugin list
wpfy sftp example.com --enable
wpfy cron status
wpfy smtp status
wpfy dns cloudflare status

Product capabilities

Four operational areas that define how WPFY runs WordPress on a VPS.

Per-site isolation

Each managed site is one Compose project with a private network, dedicated database and volumes, unique Unix UID, and unprivileged Nginx.

  • One Compose project per site
  • PHP 7.4 through 8.4 where currently supported
  • Designed to reduce cross-site blast radius
wpfy run client-a.com --wp --php 8.3
wpfy run client-b.com --wp --php 7.4 --wpredis
wpfy site status client-a.com

SSL and DNS safety

Let's Encrypt issuance runs only after DNS/IP preflight passes. Cloudflare proxy detection helps avoid failed issuance and rate-limit problems.

  • DNS/IP preflight before ACME
  • Traefik-managed certificates
  • Cloudflare-only wildcard SSL
wpfy site ssl example.com --letsencrypt
wpfy dns cloudflare set
wpfy config example.com --dns cloudflare

Backup and recovery

Verified file and database archives with local retention, S3-compatible remote lifecycle, scheduled all-site backups, and Traefik edge backup.

  • Explicit retention and prune controls
  • Remote list, restore, delete, and prune via WPFY
  • Validation before live restore mutation
wpfy backup example.com --s3
wpfy backup remote list example.com
wpfy restore example.com --latest
wpfy backup edge --s3

Operations and access

Health checks, structured debug output, container logs, WP-CLI, scoped SFTP, cron timers, and SMTP configuration with explicit test sends.

  • Sanitized config display via wpfy config
  • Scoped SFTP sidecar per site
  • Safe refresh workflows with optional restart
wpfy healthcheck all
wpfy wp example.com plugin list
wpfy sftp example.com --enable
wpfy smtp test --to ops@example.com

Quick-start workflow

Three steps from a fresh VPS to a verified managed site. Full procedures live in the documentation.

Step 1

Prepare a fresh VPS

Supported Ubuntu environment, sudo/root access, domain DNS pointing at the server, and outbound network access.

Step 2

Install and create a site

Run the installer, then create your first WordPress site with a flat command.

curl -fsSL https://raw.githubusercontent.com/wpfyorg/wpfy/main/install.sh | sudo bash
wpfy run example.com --wp
Mutable reference: The curl command above fetches from the main branch. Download and review the script first, or pin WPFY_REF and WPFY_SOURCE_SHA256. Review-first install
Step 3

Operate and verify

Confirm site state and run operator health checks before routing production traffic.

wpfy site status example.com
wpfy healthcheck all
wpfy debug example.com

Intended users

WPFY is built for operators who manage their own infrastructure — not for untrusted multi-tenant hosting without independent review.

WordPress agencies

Run client sites on one VPS with per-site isolation and scoped SFTP handoff.

Freelance developers

Provision sites with repeatable commands and keep root access to yourself.

VPS operators

Replace ad-hoc scripts with a consistent CLI for stack and site lifecycle.

Technical self-hosters

Self-host WordPress with Docker isolation and explicit backup workflows.

Know the boundaries before you install

Honest limits are part of the product. WPFY is open source under AGPL-3.0 — not a hosting company or graphical control panel.
